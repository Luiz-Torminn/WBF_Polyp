We are going to build an ensemble method based on Weighted Box Fusion for object detection. The objective is to take the final predicted bounding boxes generated by each model and fuse them into a final detection output that may improve overall detection metrics.

The models to be used are:

- RFDETR nano
- YOLOv12 nano
- DEIMv2 pico

The implementation scope is strictly focused on machine learning inference, ensemble fusion, evaluation, and reproducible experimentation for object detection. This is not a service-oriented or PRD-driven backend task. The core goal is to make the three models run consistently on the same evaluation pipeline, collect their predictions in a compatible format, apply Weighted Box Fusion correctly, and compare the ensemble results against the standalone models.

At the end, the project must contain a `main.py` that can be run directly from the CLI with `python3 main.py`. All generated outputs must go to `.outputs/` and this folder must be gitignored. ## Evaluation Threshold Semantics

The pipeline keeps two distinct thresholds, and they must not be conflated:

- **Predict / score threshold** (low, e.g. `0.001`): applied during per-model inference so the full precision-recall curve is preserved for mAP computation. This matches what `RFDETR/main.py` (`PREDICT_THRESHOLD = 0.001`) and `YOLO_model/main.py` (`conf=0.001`) already use.
- **Fusion `skip_box_thr`** (e.g. `0.0001`): a WBF-internal filter applied inside `weighted_boxes_fusion()` before fusion. It is independent of the inference threshold and is exposed as a CLI flag.

Standalone metrics in the final CSV are computed by the project's unified evaluator (pycocotools COCOeval + a precision/recall extractor at the best-F1 score point on the IoU=0.5 PR curve). The native evaluators in each upstream repository (`RFDETR/main.py`, `YOLO_model/main.py`, `DEIMv2/train.py --test-only`) remain runnable from their own directories for reference but are not the source of the CSV numbers.
