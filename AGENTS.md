We are going to build an ensemble method based on Weighted Box Fusion for object detection. The objective is to take the final predicted bounding boxes generated by each model and fuse them into a final detection output that may improve overall detection metrics.

The models to be used are:

- RFDETR nano
- YOLOv12 nano
- DEIMv2 pico

The implementation scope is strictly focused on machine learning inference, ensemble fusion, evaluation, and reproducible experimentation for object detection. This is not a service-oriented or PRD-driven backend task. The core goal is to make the three models run consistently on the same evaluation pipeline, collect their predictions in a compatible format, apply Weighted Box Fusion correctly, and compare the ensemble results against the standalone models.

At the end, the project must contain a `main.py` that can be run directly from the CLI with `python3 main.py`. All generated outputs must go to `.outputs/` and this folder must be gitignored. ### What is Success

Success means the implementation fully satisfies the requested behavior and is aligned with the real execution requirements of RFDETR nano, YOLOv12 nano, and DEIMv2 pico.

The code must work correctly in practice, not only syntactically. Each model must load successfully using the provided pretrained weights, run inference on the intended dataset, produce valid predictions, and integrate into a shared pipeline that applies Weighted Box Fusion correctly.

A successful implementation must produce:

- a runnable `main.py`
- a reproducible inference and evaluation flow
- standalone metrics for all three models
- fused ensemble metrics
- outputs written under `.outputs/`
- a descriptive CSV summary with all requested metrics

Success also means the implementation is faithful to the machine learning purpose of the task. The fusion process must combine model outputs in a technically correct way for object detection, respecting prediction format compatibility, score handling, label alignment, and box normalization requirements. The ensemble must not be a superficial wrapper around incompatible predictions.

A successful implementation is also observable and maintainable. Logs, intermediate outputs when useful, configuration points, and evaluation behavior should be clear enough to support debugging and future experimentation. The code should make it easy to inspect whether performance gains actually come from valid ensemble behavior.

### What is Fail

Failure means the implementation only appears to work superficially or partially, while violating the actual requirements of the model repositories, the requested ensemble behavior, or the practical evaluation flow.

Examples of failure include:

- one or more models do not run correctly with their documented or actual inference logic
- pretrained weights are not loaded correctly
- outputs are fused using inconsistent box formats
- labels or scores are mismatched across models
- WBF is applied incorrectly
- the ensemble runs but compares unfairly against standalone models
- metrics are produced from inconsistent evaluation settings
- the final pipeline works only on a narrow happy path and is not reproducible

Failure also includes cases where the implementation does not truly serve the object detection ensemble objective. For example, if the code runs but ignores important repository-specific preprocessing or post-processing constraints, or if it introduces hidden assumptions that make the fusion invalid, it must be treated as failed.

An implementation must also be considered failed if validation is incomplete. If only code execution was checked, but not real inference behavior, prediction compatibility, metric generation, or end-to-end pipeline integrity, then the task is not fully validated. “It runs” is not enough. The implementation must be proven correct for the intended machine learning workflow.
